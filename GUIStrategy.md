# GUI Strategy — Closing the UX Gaps
**Document type:** Implementation strategy & execution plan  
**Companion to:** [GUI_UX_AUDIT.md](GUI_UX_AUDIT.md)  
**Author role:** Principal UI/UX Architect & MLOps Product Manager  
**Date:** 2026-05-18  
**Target outcome:** A v1.0-production-ready Streamlit GUI in ≤4 development days, with a clear path to v1.x quality-of-life upgrades.

---

## 0. Guiding Principles

Before diving into sequencing, the following five principles inform every decision below:

1. **Stop the bleed first.** Gaps that cause silent data corruption (M-05, M-06) or that strand artifacts on disk (M-01) outrank cosmetic polish — they are landed in the first 24 hours.
2. **Native Streamlit only.** No custom React components, no `streamlit_extras` heavy dependencies. Every fix proposed in the audit can be expressed with `st.dataframe`, `st.fragment`, `st.download_button`, `st.expander`, `st.components.v1.html` or `st.column_config`. This keeps the build reproducible across Streamlit 1.31+ and avoids version-lock risk.
3. **One shared helper > ten ad-hoc fixes.** Wherever multiple audit items collapse onto the same root cause (e.g., "no download buttons anywhere"), build one helper module rather than scatter copies. See §3.
4. **State is a schema, not a junk drawer.** Today's `_STATE_DEFAULTS` dict is the only definition of session state. We will treat it as an explicit, typed, versioned schema. Each new key gets a docstring entry and a migration rule when shape changes. This is what makes W-01/W-03/W-04 fixable without regression.
5. **Every gap closed must be testable.** "Works on my machine" is not a definition-of-done. §6 specifies the test layer.

## 1. The Three-Phase Roadmap

The 27 audit items fall naturally into three phases. Each phase has a fixed scope, a fixed exit criterion, and a fixed budget. We do not advance to the next phase until the current one is shipped and observable.

### Phase 1 — Stop the Bleed (Day 1, ~6 hrs)
Goal: Eliminate silent data loss and the two CRITICAL items. After this phase, no GUI interaction can cause a manifest to be silently mis-built or an artifact to be unrecoverable from the UI.

| Item | Effort | Why first |
|------|--------|-----------|
| M-05 (stale pair-params) | 30 min | One-line filter in `build_run_manifest`. Silent corruption. |
| M-06 (data_editor key collision) | 45 min | One-line counter increment. Silent corruption. |
| M-01 (artifact downloads) | 2 hr | CRITICAL. Single largest user complaint vector. |
| M-08 (preflight error display) | 1 hr | High-frequency error path; today it's an opaque DataFrame. |
| M-07 (FAILED-run drill-down) | 1.5 hr | Pairs naturally with M-08. |

**Exit criterion:** A new engineer can register a dataset, run a job that fails preflight, see the structured error, fix the manifest, re-run, and download the resulting `metrics.json` and `embeddings.h5` — all without opening a terminal.

### Phase 2 — Foundation (Days 2–3, ~12 hrs)
Goal: Make the GUI survive the "500-runs test" and feel responsive at enterprise scale.

| Item | Effort | Depends on |
|------|--------|------------|
| M-02 (results pagination) | 1.5 hr | Phase 1 |
| M-03 (matrix scalability) | 4 hr | Phase 1 |
| M-04 (log truncation + download) | 1 hr | M-01 helper |
| M-09 (save/load manifest) | 3 hr | Session-state schema (§4) |
| M-10 (registry-dirty banner) | 30 min | — |
| S-04 (fragment-based log loop) | 2 hr | — |

**Exit criterion:** Synthetic load test (100 datasets, 50 models, 1,000 runs — fixtures generated by §6 test harness) loads every tab in <3 s and no widget interaction triggers >500 ms of frontend lag.

### Phase 3 — Polish & Onboarding (Day 4, ~6 hrs)
Goal: First-time users reach a successful run without external guidance.

| Item | Effort |
|------|--------|
| O-01 + N-09 (welcome hero + example dataset button) | 2 hr |
| O-02/O-03 (cross-tab navigation shortcuts) | 1 hr |
| O-04 + N-01 (sidebar workflow stepper) | 1.5 hr |
| O-05, S-06, W-06 (tooltip + width + canonical exp name) | 1 hr |
| E-05/E-06 (registration stderr + error-event grouping) | 30 min |

**Exit criterion:** New-user usability test (recruit 2 colleagues unfamiliar with the tool, give them a sample H5AD, observe them reach a successful run). Time-to-first-run ≤10 minutes with zero verbal prompting.

### Deferred to v1.x — Nice-to-Haves
N-02 (cross-run comparison), N-03 (embedding preview), N-04 (full manifest import), N-05 (dark mode), N-06 (native Optuna browser), N-07 (timeline), N-08 (quick launch), N-10 (log filter), A-04 (link columns), A-05 (compare runs), A-06 (native MLflow artifact API), E-07 (live path validation), S-05 (MLflow batch fetch), W-04 (cross-session run lock).

These are tracked as individual issues but explicitly **not** scheduled. They get done when (a) a user file an issue, or (b) Phases 1–3 land with leftover budget.

## 2. Sequencing Rationale

The 10 Must-Have items are reordered from the audit's "by impact" list into a **dependency-aware** order. The audit listed `M-01 → M-02 → M-07 → M-04 → M-03 → M-05 → M-06 → M-08 → M-09 → M-10`; the strategy reorders to:

```
M-05 → M-06 → M-08 → M-07 → M-01 ║ M-02 → M-03 → M-04 → M-09 → M-10
       Phase 1                       Phase 2
```

Two reasons:

1. **Cheapest correctness fixes first.** M-05 and M-06 are 30–45 minute one-liners that prevent silent data corruption. They land before *any* visible feature work because they make subsequent QA reliable — there is no point user-testing a manifest builder that silently corrupts manifests.
2. **Error-path before happy-path.** M-08 (preflight errors) and M-07 (FAILED-run drill-down) ship before the new artifact UI (M-01) because as users start exercising the GUI more, the volume of preflight failures rises sharply. We want the error-display infrastructure in place *before* increased traffic exposes it.

## 3. Shared Infrastructure to Build Once

Three new modules collapse most of the audit items into one-line consumers. Building these first is what makes the timeline credible.

### 3.1 `multiverse/gui_artifacts.py` — Artifact rendering helpers
Closes: M-01 (A-01, A-02, A-03), M-04, A-06 (deferred but stub here), N-10 partial.

```python
def render_artifact_tree(artifact_dir: Path, *, max_inline_mb: float = 200) -> None: ...
def render_log_viewer(log_path: Path, *, default_tail: int = 200, with_filter: bool = True) -> None: ...
def render_download_button(path: Path, label: str | None = None) -> None: ...
```

Every place that currently shows a path string (`output_path`, container.log path, MLflow artifact dir) becomes a one-line call to these helpers. This is also where Phase-3's log filter (N-10) lives — `render_log_viewer` gets the `search` text_input added once and inherits to all callers.

### 3.2 `multiverse/gui_state.py` — Typed session-state schema
Closes: W-01, W-03, W-04, W-06, and prevents 80% of future bugs.

```python
@dataclass
class GuiState:
    selected_datasets: list[str]
    selected_models: list[str]
    planned_jobs: list[PlannedJob]
    run_mode: Literal["Use User Params", "Run Gridsearch"]
    registry_dirty: bool
    editor_version: int                # NEW — used by M-06
    experiment_name: str               # NEW — canonical, replaces jb_exp_name + exec_live_exp_name
    is_running: bool
    pending_launch: dict | None

def init_state() -> GuiState: ...
def get_state() -> GuiState: ...        # validates schema on read
def bump_editor_version() -> None: ...
```

Backing store remains `st.session_state` — the dataclass is a *view*. This lets us add fields without re-keying widgets, and to add migration hooks (handle a v1 layout → v2 in one place).

### 3.3 `multiverse/gui_navigation.py` — Cross-tab navigation
Closes: O-02, O-03, partial O-04, partial N-01.

Streamlit doesn't expose programmatic tab switching. The proven workaround is `st.query_params` + a top-level dispatch:

```python
TABS = ["registry", "jobs", "params", "execute", "results", "mlflow", "optuna", "settings"]

def go_to(tab_slug: str) -> None:
    st.query_params["tab"] = tab_slug
    st.rerun()

def current_tab_index() -> int:
    return TABS.index(st.query_params.get("tab", "registry"))
```

The `st.tabs(...)` call in `main()` consumes `current_tab_index()` to choose the default. Every "Go to X →" button is one `go_to("x")` call. This also gives us deep-linkable URLs for free (a v1.x bonus that we get without paying for).

## 4. Concrete Architectural Patterns

### 4.1 Fragment-First for Live Updates
Today's `_render_execute_tab()` polling loop is the single biggest performance liability (S-04). Strategy: wrap the streaming subprocess panel in an `@st.fragment(run_every=timedelta(seconds=1))`. The fragment owns:
- The status table placeholder
- The log placeholder (`deque(maxlen=1000)` for line storage)
- The cancel button (stable key, not index-incremented)

The full page reruns only on tab switch or user interaction — not on every 500 ms log line. This is also the right home for M-04's "show last N lines" control: it rerenders cheaply inside the fragment.

### 4.2 Pagination Pattern (M-02)
A reusable `paginate(query_fn, page_size=50)` helper used by Results today and Run-History (v1.x N-07) tomorrow:

```python
def paginate(total_count_fn, page_fn, page_size=50, key="page"):
    total = total_count_fn()
    n_pages = max(1, math.ceil(total / page_size))
    page = st.number_input("Page", min_value=1, max_value=n_pages, value=1, key=key)
    rows = page_fn(limit=page_size, offset=(page - 1) * page_size)
    st.caption(f"Page {page} of {n_pages} · {total} total rows")
    return rows
```

This is what makes M-02 a 30-minute change rather than a 90-minute one, and N-07 (timeline) trivially paginatable later.

### 4.3 Compatibility Matrix Reduction (M-03)
The 100×50 = 5,000-cell editor is unsalvageable at scale. The strategy is **never render the full matrix**; render a "selection-first" UI:

1. Top: `st.multiselect("Datasets")` + `st.multiselect("Models")` with sensible defaults (all selected up to 10×10, empty above).
2. Middle: A *display-only* `st.dataframe` color-coded compatibility matrix limited to the selected subset.
3. Bottom: The editable `data_editor` is also limited to the subset. Switching `Selected` checkboxes only operates on visible rows.

For the "Select all compatible" use case (which is the implicit reason the current UI shows everything), add a "Auto-select all compatible pairs from filtered subset" button. This preserves the workflow without the 5,000-cell DOM tree.

### 4.4 Error Schema (M-08)
The audit notes that `parsed_manifest.errors` has an "unknown" structure. Step 0 of M-08: read `multiverse/runner/cli.py` and document the actual schema. If it's already structured (`code`, `field`, `message`), use it directly. If it's stringly-typed, this is the moment to refactor — exactly one place reads it (the GUI) and exactly one place writes it (the validator), so the change is local. The decision belongs in this implementation phase, not deferred.

### 4.5 Save/Load Manifest (M-09)
Save is trivial — change one hardcoded path to a text_input. Load is the interesting half. The strategy:

- **Phase 2 (Must-have):** Load only sets `experiment_name`, `random_seed`, `run_mode`. These three keys do not require widget pre-population — they are read at manifest-generation time.
- **Phase 3+ (N-04):** Full state reconstruction (auto-check the data_editor, pre-fill hyperparam widgets) requires writing to widget keys before they render. Streamlit supports this via `st.session_state[widget_key] = value` *before* the `st.text_input(key=widget_key)` call. The reason it's deferred to N-04 is that we need to first ship M-06 (editor_version) — otherwise loading a manifest into a stale data_editor reintroduces the silent-selection bug.

This ordering — M-06 before N-04 — is non-obvious but critical and is the kind of dependency the audit alone doesn't surface.

## 5. Risk Register

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| `st.fragment` requires Streamlit ≥1.33 — repo may be on older version | Low | High (S-04 blocked) | Pin `streamlit>=1.33` in `pyproject.toml` as part of Phase 1. If breaking changes, fall back to `st.empty()` placeholder pattern (no auto-rerun, manual `st.rerun()`). |

| Phase-3 onboarding requires a sample dataset bundled in the repo, but datasets are large | High | Low | Use a 50-cell synthetic AnnData (~50 KB) generated by `scanpy.datasets.pbmc3k().subset(50)` cached as a fixture. Acceptable for tutorial purposes. |

## 6. Testing Strategy

Frontend correctness is what we ship — so it must be testable. Three layers:

### 6.1 Unit-level (`tests/unit/test_gui_helpers.py`)
The pure functions (`build_run_manifest`, `slugify_experiment_name`, `_parse_memory_gb`, `_wave_simulate`, `_flatten_dict`) get exercised by pytest. M-05's fix lives here — a one-line test asserts that pair_params for unselected jobs do not appear in the output manifest.

### 6.2 Component-level (`tests/gui/test_streamlit_app.py`)
Streamlit ships an official testing framework: `streamlit.testing.v1.AppTest`. It runs the app in-process and lets you assert against rendered widgets without a browser.

```python
def test_results_pagination():
    at = AppTest.from_file("multiverse/gui.py").run()
    at.session_state["results_page"] = 2
    at = at.run()
    assert "Page 2 of" in at.caption[0].value
```

Phase-1 deliverable: at least one AppTest per Must-Have item.

### 6.3 Scale fixtures (`tests/fixtures/synthetic_registry.py`)
A small Python script that populates `mvexp_state.db` with N datasets, M models, K runs. Run before manual QA of Phase 2's scalability fixes. Reproduces the "500-runs test" locally in ~3 seconds.

### 6.4 Manual usability test (Phase 3)
Two unfamiliar colleagues, sample dataset, observe-only. Time-to-first-run + verbal think-aloud. Cheap, irreplaceable.

## 7. Telemetry & Definition of Done

### 7.1 Lightweight Telemetry
Add one helper, `gui_telemetry.track(event_name, **fields)`, that appends to a local JSONL log at `~/.multiverse/gui_events.jsonl`. Events to instrument:
- `tab_switched` (tab name, from-tab, timestamp)
- `manifest_generated` (n_jobs, run_mode)
- `run_launched` (n_jobs, manifest_path)
- `run_completed` (status, duration_s)
- `error_shown` (component, error_kind)
- `download_clicked` (artifact_kind, size_mb)

No network calls. Pure local. Lets us answer "is anyone actually using the artifact download?" in a v1.x retro. Opt-out via env var `MULTIVERSE_GUI_TELEMETRY=0`.

### 7.2 Definition of Done — Per Item
A Must-Have is "done" only when **all five** are true:
1. Code change merged.
2. AppTest covering the new path passes.
3. Manual smoke test against the synthetic-registry fixture passes.
4. `GUI_UX_AUDIT.md` row updated with strikethrough + commit SHA.
5. Telemetry event added if the change introduces a new user-visible action.

### 7.3 Definition of Done — Per Phase
- **Phase 1:** All five Must-Haves merged; synthetic fixture confirms no silent corruption; artifact downloads work for the smallest and largest fixture run.
- **Phase 2:** Synthetic 100×50×1000 fixture: every tab renders in <3 s, no interaction blocks the main thread >500 ms.
- **Phase 3:** Two new-user observations completed; time-to-first-run ≤10 min in both.

## 8. What This Strategy Explicitly Does *Not* Do

To prevent scope creep, the following are out of scope for v1.0:

- **No new backend changes.** The audit's premise was "assume the backend works." If a Must-Have surfaces a backend gap (e.g., `failure_reason` schema in M-08), we minimally adapt the backend and document — we do not redesign.
- **No design-system overhaul.** No custom color palette, no logo, no marketing polish. Streamlit defaults until v1.x.
- **No mobile responsiveness.** Streamlit's mobile support is poor; users on phones can wait for v1.x or use the MLflow UI.
- **No internationalization.** English only.
- **No role-based access control.** Single-user assumption holds for v1.0; multi-tenant is a v2 conversation.
- **No new Streamlit dependencies.** `streamlit-aggrid`, `streamlit-extras`, `streamlit-elements` are all interesting and all explicitly rejected for v1.0 to keep the dependency surface tight.

## 10. Summary

The audit identified 27 gaps; this strategy resolves them via 3 shared helpers, 10 Must-Have fixes, and 14 deferred Nice-to-Haves. Effort: **~24 dev hours over 4 calendar days.** The plan is dependency-aware (M-06 before N-04), risk-aware (Streamlit version pinning, OOM guards on artifact downloads), and testable end-to-end (`streamlit.testing.v1.AppTest` + synthetic fixtures).

The single most important architectural commitment is **§3's shared-helper-first approach**: by building `gui_artifacts.py`, `gui_state.py`, and `gui_navigation.py` as the first concrete artifacts of Phase 1, every subsequent Must-Have becomes a 5–15 line consumer rather than a from-scratch implementation. This is what makes the 4-day timeline credible.
