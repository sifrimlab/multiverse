# Critique of Production Readiness Audit

**Posture:** Senior Red Team Engineer + MLOps Auditor  
**Target:** `PRODUCTION_READINESS_AUDIT.md`  
**Scope correction:** mvexp is a local, single-user workstation tool. That does not excuse weak durability semantics. It means the platform must be brutally clear about what it is: a local research appliance, not a multi-user service. The original audit still leaves dangerous optimism around SQLite, filesystem promotion, recovery, storage pressure, partial success, and the amount of control-plane machinery being pushed onto rookie users.

| Audit Finding | Is it sufficient? | Why it fails at scale/complexity | Principal-Level Improvement |
|---|---|---|---|
| "Single-user local install" is used to de-scope multi-user contention. | No. It is a valid product boundary, but the audit lets that boundary become a reliability shortcut. | A single local user still creates concurrent writers: Streamlit, runner subprocesses, CLI registration, MLflow, Optuna, and Docker callbacks. "Single user" is not "single process." If 10 researchers share one GUI, it fails immediately as an unsupported deployment model; worse, the docs do not make that a hard boundary. | Declare multi-user access unsupported in the product contract. Add startup guardrails that bind only to localhost by default, reject shared network hosting, and document the migration path: server API, Postgres, object storage, and a real job queue. |
| SQLite is treated as the source of truth. | Partially. It is acceptable for a local appliance, but not as the only truth. | SQLite can be corrupted, truncated, locked by antivirus/indexers, left with stale WAL files, or sit on a flaky network-mounted home directory. The audit discusses `database is locked`, but not DB file corruption, WAL recovery failure, schema migration interruption, or mismatches between DB rows and artifact reality. | Treat SQLite as an index, not the sole truth. Store immutable per-run manifests, state transition journals, and result metadata inside artifact bundles. Add `PRAGMA integrity_check`, backup/restore, WAL checkpoint policy, schema migration transactions, and a rebuild-index command from the artifact store. |
| WAL mode + busy timeout is described as enough for concurrency. | No. WAL reduces reader/writer blocking; it is not a coordination system. | The runner actor serializes writes only inside the runner process. GUI writes and CLI writes bypass that actor. A long GUI registration transaction can still block runner finalization. A timeout still becomes a user-facing failure during promotion or metric persistence. | Move all state mutations behind one local control-plane service or explicit file lock. If staying embedded, use short transactions, external lock files around migrations/registration, retry policies with bounded jitter, and state operations designed to be idempotent after retry. |
| Single-writer DB actor is proposed as a hardening pattern. | Not enough. It is a process-local actor, not a system actor. | If the actor crashes, the runner may continue containers whose state cannot be persisted. If its queue backs up, completed runs wait on a single bottleneck. If the whole Streamlit-launched runner dies, the actor dies with it. No external lease or durable queue preserves intent. | Replace the in-process actor with a local supervisor/job daemon, or persist write intents to an append-only journal before side effects. Actor restarts must replay uncommitted intents and reconcile Docker/container/artifact state. |
| Promotion bug is identified around `shutil.move` and marker files. | No. It names the symptom but underspecifies the fix. | The audit says "make promotion crash-safe" but does not require a full saga. Promotion touches DB state, workspace path, artifact path, marker file, metrics, MLflow, and container cleanup. Any partial ordering can create split-brain truth. | Implement idempotent promotion as a saga: `prepare -> validate -> fsync -> atomic rename or staged copy -> commit manifest -> DB commit -> external tracking`. Each step must be replayable. Never delete a destination unless the current run owns it via a durable ownership token. |
| Cross-filesystem `shutil.move` is flagged. | Incomplete. | Refusing `EXDEV` is safer than silent copy/delete, but a local user will naturally put workspaces on fast scratch and artifacts on large storage. Refusal alone turns a reliability issue into a usability wall. | Support two modes explicitly: same-filesystem atomic rename, or managed copy promotion with checksums, size manifests, fsync, temp destination, and atomic final directory swap. Surface which mode is active before launch. |
| Recovery deletes incomplete promotion output when marker is absent. | No. The audit stays at the data-loss symptom and misses the deeper principle. | Recovery code should be the most conservative code in the system. Here it can destroy the only successful output. The audit does not demand a "never delete unknown data" invariant. | Recovery must quarantine, not delete. Move suspicious directories to `store/quarantine/`, write a recovery report, and allow user-assisted adoption or deletion. Deletion should require explicit GC policy and ownership proof. |
| Evaluation failures after training are noted as false `SUCCESS`. | No. It proposes a binary repair for a non-binary workflow. | The audit frames this as "mark failed instead." That may be too blunt. In science workflows, model training success and evaluation failure are different facts. Collapsing both to `FAILED` hides partial value and can trigger wasteful reruns. | Introduce explicit terminal/subterminal states: `TRAINING_SUCCEEDED`, `EVALUATION_FAILED`, `PROMOTION_FAILED`, `TRACKING_FAILED`, `SUCCESS`. Do not inject final metrics metadata until evaluation is atomically complete. Persist model-native outputs separately from benchmark metrics. |
| Partial successes are not modeled. | No. This is one of the largest missing edge cases. | A run can produce embeddings but fail UMAP, produce metrics but fail MLflow upload, train 9 of 10 jobs, or finish training while evaluation crashes due to `scib-metrics`. The current green/red model is too coarse for reproducible research. | Use artifact-scoped completeness manifests. Each artifact declares `produced`, `validated`, `checksum`, `producer`, and `timestamp_source`. GUI should show partial completion explicitly and block leaderboard comparison unless evaluation completeness is satisfied. |
| MLflow failure is allowed to continue after promotion. | Not sufficient. | If MLflow is advisory, the audit should say so and prevent users from treating MLflow as canonical. If MLflow is part of reproducibility, continuing silently creates split-brain results between `store/` and dashboards. | Define MLflow as either cache or contract. If cache: add a repair/sync command from artifacts. If contract: make tracking a state transition with retry and explicit `TRACKING_FAILED`. |
| Pre-flight validators are proposed heavily. | Risky. | More validators can become configuration hell: rookies get blocked by a wall of schema errors before running PCA on one dataset. Worse, validators drift from container reality and reject valid workflows or bless invalid ones. | Split validation into levels: `basic` default checks that prevent obvious waste, `strict` reproducibility checks for publication, and `developer` contract checks for new models. Validators must be versioned and tied to model manifests, not scattered through the runner. |
| "Deepen pre-flight" is recommended as a broad fix. | Over-engineered if applied indiscriminately. | Biological data is messy. Over-validating `obs`, categorical fields, count-likeness, and modality assumptions can turn mvexp into a data curation platform rather than a model execution platform. | Keep the core platform thin. Provide warnings and explainability by default, hard-fail only on invariants required by the selected model contract. Push domain-specific checks into model-specific validators. |
| Docker worker orchestration is treated as local process management. | Incomplete. | Docker daemon restart, Docker Desktop sleep, rootless Docker quirks, image prune, container name collisions, and daemon unavailability can all split the runner's belief from actual container state. | Add a local supervisor with leases. Every launched container should have labels linking it to run ID, manifest hash, workspace, and owner. Reconciliation should query Docker by label and repair state, not rely only on stored container IDs. |
| GUI subprocess control is identified as weak. | No. It underestimates Streamlit's control-plane mismatch. | Streamlit reruns are a poor owner for long-lived jobs. Session state is not a durable job server. Browser lifecycle and job lifecycle are fundamentally different. | Rewrite GUI execution around a local daemon/API. The GUI submits jobs, polls job state, and sends cancel requests. The daemon owns subprocesses, Docker containers, DB writes, and recovery. |
| Cancellation is proposed as PID tracking plus SIGTERM/SIGKILL. | Not enough. | Killing the Python runner does not guarantee child containers stop, MLflow finalizes, workspace is marked cancelled, or partial outputs are quarantined. | Cancellation must be a saga too: mark `CANCEL_REQUESTED`, stop containers by Docker label, wait grace period, snapshot logs, mark terminal `CANCELLED`, quarantine or preserve workspace according to policy. |
| Disk-pressure circuit breaker is recommended. | Incomplete. | Checking `store/` at launch does not prevent disk filling during run, Docker image pulls, logs, MLflow artifacts, Optuna DB growth, or cross-device copy promotion. | Implement continuous storage accounting: per-run predicted size, live free-space checks, Docker data-root checks, log rotation, and promotion reserve space. Stop admission before both `store/` and Docker data-root are dangerous. |
| Garbage collection is only implied. | No. | Workspaces, failed runs, Docker containers, images, logs, MLflow artifacts, Optuna studies, WAL files, and quarantine directories will grow until a local disk fills. This is guaranteed on a research laptop. | Add first-class GC: retention policy, dry-run preview, per-run ownership metadata, "delete failed workspaces older than N days", "archive successful artifacts", and "prune Docker objects created by mvexp labels." Never GC without ownership labels/checksums. |
| Read-only or degraded `store/` mount is not discussed. | No. | A user can run from an external drive, institutional network mount, synced folder, or disk that becomes read-only after filesystem errors. SQLite may still read while artifact writes fail. Promotion, logging, and recovery then cascade. | At startup and before each run, perform capability probes: create, fsync, rename, chmod/chown expectation, available inode count, and write/delete in workspace/artifact/temp directories. Fail closed before launching containers. |
| Database corruption is not discussed. | No. | SQLite is robust, not magic. Power loss, network filesystems, broken sync clients, or manual edits can corrupt the DB. Without rebuild, the platform becomes unrecoverable or lies about prior runs. | Add `mvexp doctor` and `mvexp rebuild-index`. The artifact store should contain enough manifests and metadata to reconstruct `datasets`, `models`, `runs`, and `run_metrics` with conflict reports. |
| Clock skew is ignored. | No. | Local host and Docker containers can disagree because of VM time sync, sleep/resume, Docker Desktop, or manual clock changes. Timestamps in logs, MLflow, Optuna, artifacts, and DB can sort incorrectly, breaking forensic reconstruction. | Use monotonic durations for ordering and wall-clock only for display. Record host timestamp, container timestamp, monotonic start/end, timezone, and clock source. Do not use timestamps as correctness keys. |
| Artifact metadata injection has no atomic boundary. | No. | If metrics persistence succeeds but artifact writing fails, or `run_metrics` is updated from stale `metrics.json`, the GUI can display a result that is not actually present in the artifact bundle. | Make metadata injection content-addressed. Load metrics from an immutable artifact manifest with checksums; update DB in one transaction with the artifact manifest hash. If the hash changes, require re-indexing. |
| Result identity relies on generated run IDs. | Weak. | Two identical manifests create unrelated runs, while a rerun after partial failure may overwrite or collide depending on path construction. The audit mentions replay but not content addressing. | Use manifest hash + dataset hash + model image digest + params hash as logical run identity. Generated IDs can remain physical attempts. Separate "experiment attempt" from "scientific recipe." |
| Model image tags are treated as reproducibility metadata. | Insufficient. | Tags are mutable. A local user can rebuild `multiverse-pca:1.0.0` and silently change behavior. The audit mentions image digest but does not make it a hard contract. | Pin and record image digest at execution. Refuse publication/strict mode if digest is missing. Store container SDK version, git commit, dependency lock hash, and model manifest hash in the artifact manifest. |
| Container outputs are assumed trustworthy if required files exist. | Too optimistic. | A model can write malformed HDF5, wrong latent shape, NaN embeddings, mismatched cell order, or metrics for a different dataset. File existence is not validation. | Validate output semantics before promotion: row count equals input cells, stable cell IDs match, finite numeric embeddings, metrics schema validates, and all required artifacts are checksummed. |
| "Required artifacts were written, validated, and promoted" appears in docs. | False as currently described. | Evaluation happens after success in the inspected runner path, and validation appears weaker than the docs imply. This creates dangerous trust in the green status. | Align docs and state machine with reality. Either perform validation before terminal success or rename statuses so users know what was actually completed. |
| Configurable storage root is recommended. | Necessary but dangerous if naive. | Letting rookies point storage anywhere creates new failures: paths under Dropbox/OneDrive, network filesystems, FAT/exFAT without POSIX semantics, read-only institutional mounts, or locations not mounted into Docker. | Storage config must be validated as a storage backend, not accepted as a string. Probe filesystem semantics, Docker mount visibility, free space, case sensitivity, symlink behavior, and permission ownership. |
| Port-collision checks are recommended. | Useful but not enough. | Local ports are only one service dependency. MLflow can be reachable but configured with unusable artifact roots. Optuna dashboard can show stale studies. Docker can be reachable but wrong context. | Add service health checks that verify end-to-end behavior: create temp MLflow run and artifact, create temp Optuna study, launch tiny Docker container with mounted `/input` and `/output`, then clean up. |
| Root-owned output files are identified. | No. The scope is too narrow. | UID/GID handling is more than convenience. It affects cleanup, GC, artifact downloads, Jupyter analysis, and whether Streamlit can render outputs after container writes. | Standardize container user mapping. Run containers with host UID/GID where possible, validate write ownership during pre-flight, and record ownership expectations in `mvexp doctor`. |
| OOM accounting is flagged as static. | Incomplete. | Memory is only one resource. CPU oversubscription, GPU VRAM, file descriptors, inode exhaustion, Docker log growth, and MLflow DB locks also kill local runs. | Introduce a local resource broker with live observation for RAM, VRAM, disk, inodes, and Docker daemon health. Surface resource denial as a clear scheduling decision, not a late model failure. |
| Retry is recommended for transient failures. | Dangerous without idempotency. | Retrying image pulls is fine. Retrying promotion, MLflow logging, DB writes, or evaluation can duplicate metrics, overwrite artifacts, or mark the wrong attempt successful. | Only retry operations with idempotency keys. Every external side effect needs a deterministic key: run attempt ID, artifact manifest hash, MLflow run ID, Optuna trial ID, and Docker labels. |
| Operator runbook is proposed. | Needed, but not sufficient for rookies. | A rookie user will not read a runbook while a laptop is hot and disk is full. The system must provide diagnosis in-app. | Build `mvexp doctor` into the GUI: DB health, Docker health, storage health, orphaned containers, orphaned workspaces, stale WAL, MLflow artifact proxy, and suggested one-click safe repairs. |
| The audit treats "more guardrails" as mostly positive. | No. Guardrails can become failure modes. | Validators, pre-flight checks, state machines, actors, repair logic, and service dashboards add configuration and cognitive load. Each guardrail can block valid work, drift from reality, or create a new partial-failure edge. | Define a minimal reliable core: run container, capture outputs, validate artifact bundle, promote atomically, index results. Everything else should be optional, observable, and recoverable. |
| Registry/Store split is accepted without challenging consistency. | Weak. | Store and registry can diverge constantly: deleted artifact dirs with live DB rows, copied artifact dirs unknown to DB, edited manifests, stale model manifests, and relocated datasets. | Make Store self-describing. Every artifact dir should carry `artifact_manifest.json` with hashes and state. Registry rows should be rebuildable cache entries pointing at content-addressed store objects. |
| No explicit backup/export strategy. | Missing. | A local workstation tool is more exposed to accidental deletion, OS reinstall, laptop loss, and cloud-sync conflicts than a managed server. The audit does not ask how a researcher preserves a paper's evidence. | Add `mvexp export-run` and `mvexp export-study` that produce portable archives with manifests, checksums, metrics, logs, environment metadata, and README. Add restore/import tests. |
| Security is mostly de-scoped because local. | Too casual. | Local Docker socket access, root containers, path traversal in manifests, symlink attacks in `store/`, and arbitrary model Dockerfiles can still destroy a user's machine or data. | Treat third-party model registration as untrusted code. Validate paths stay under allowed roots, reject symlink escapes during promotion/GC, label dangerous Docker privileges, and separate "trusted local models" from imported models. |
| Complexity bloat is not called out directly. | Missing. | mvexp risks turning "compare embeddings across a few biological models" into a distributed systems platform: DB actors, sagas, dashboards, MLflow, Optuna, Docker, Streamlit, recovery, telemetry, validators, and GC. That complexity may exceed the user's actual task. | Choose a product line: local simple mode first. A single command should run a manifest and produce a portable artifact bundle without requiring MLflow, Optuna, or GUI services. Advanced services should enhance, not define, the core workflow. |

## Final Verdict

The single most fragile component to rewrite before calling this "Production Grade" is the **GUI-launched runner/control plane boundary**.

The promotion bug is severe, but it is a symptom. The deeper failure is that Streamlit, a subprocess runner, an in-process SQLite writer actor, Docker containers, filesystem promotion, evaluation, and MLflow are all loosely coupled without a durable owner. There is no authoritative local supervisor that owns job intent, leases, cancellation, recovery, idempotency, and reconciliation.

Rewrite this as a small local job daemon before adding more guardrails. The daemon should own the state machine, durable write-ahead job journal, Docker labels, cancellation saga, artifact promotion saga, and recovery logic. The GUI should become a client. Until that boundary exists, every new validator, retry, cleanup routine, or dashboard integration adds another place where partial success can become false success or silent data loss.
